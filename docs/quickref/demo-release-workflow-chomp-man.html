The release workflow — pgai-chomp-man

Drop a requirement, walk away, come back to a tested, tagged software release.

This walkthrough builds an actual game — Chomp Man, a maze arcade game whose pursuers are dragons, not ghosts — one release at a time, fully autonomously. You drop a requirement; the agent chain plans it, writes the code, tests it, and ships a tagged release on a git branch. Every command is copy-paste ready and scoped to the managed demo project pgai-chomp-man.

1 · How the kanban uses workflows

A project declares a workflow type in its project.cfg. The same agent chain — PM, CODER, WRITER, TESTER, CM — runs every project, but the lifecycle they follow is set by the workflow type. The kanban ships two:

release this demo

Software. Drives a git release-candidate branch lifecycle: open an RC → code → test → ship + tag. The managed demo project is pgai-chomp-man.

document

Prose deliverables. Drives off the requirement's version number and publishes a versioned artifact file. The managed demo project is pgai-three-bears.

You drive both the same way: drop a requirement file, and the chain runs on its own. The operator's whole job is dropping requirements and occasionally glancing at the dashboard.

2 · The release workflow, specifically

In the release workflow, each minor version is built on its own release-candidate (RC) branch, and CM is the only agent that touches the remote. One requirement becomes one RC:

PM decompose → CM open RC branch → CODER implement → TESTER verify → CM ship: merge + tag

Three things to know about how work enters a release project:

  • Requirements (vX.Y.0-*.md) are features — each minor version is a new RC.
  • Bugs (BUG-NNNN-*.md) ride the patch lane — a fix ships as the next patch.
  • Priorities (PRIORITY-NNNN-*.md) are enhancements that ride into a minor.

pgai-chomp-man runs in an isolated branch lane: branch_prefix = ai_, so its branches and tags are ai_main, ai_rc/vX.Y.Z, and tags ai_vX.Y.Z — your own main stays untouched. Each RC branches from ai_main and squashes back into it in one hop. The whole RC lifecycle, branch and tag mechanics included, runs without you.

3 · The example: build a maze game, release by release

We'll build Chomp Man up one minor at a time — each a self-contained, shippable release — then fix a bug and add an enhancement. This is the full arc that exercises every release-workflow lane:

v0.0.1
Project bootstrap — runnable pygame window skeleton
v0.1.0
Screen & movement
v0.2.0
Dots & scoring
v0.3.0
Dragons (the pursuers)
v0.4.0
Power pellets
v0.5.0 → v0.8.0
fruit bonus · sprites · warp tunnels · dragon personalities
BUG-0001
movement requires repeated keypress (should hold-to-move)
PRIORITY-0001
sound effects

Each is a single Markdown file. The whole demo is dropping these in order and watching the releases tag themselves.

4 · Run it

1

Drop the bootstrap requirement

Deposit v0.0.1 into the project. intake.sh routes by filename — the vX.Y.* prefix sends it to requirements/.

copy · pastescripts/intake.sh --project pgai-chomp-man \ v0.0.1-bootstrap-20260602.md

On the next cron wake, PM decomposes it and CM opens the RC branch ai_rc/v0.0.1. From here it's hands-off.

6

Keep shipping — features, a bug, an enhancement

Stack the rest. Features are minors; a bug rides the patch lane; a priority is an enhancement. Same intake.sh for all three — the filename decides the lane.

copy · paste# next feature minor (a new RC) scripts/intake.sh --project pgai-chomp-man v0.2.0-dots-and-scoring-20260602.md # a bug — routes to bugs/, ships as the next patch scripts/intake.sh --project pgai-chomp-man BUG-0001-movement-requires-repeated-keypress-not-held.md # an enhancement — routes to priority/, rides into a minor scripts/intake.sh --project pgai-chomp-man PRIORITY-0001-sound-effects.md

The Active-RC gate serializes them — one RC fully ships (merged + tagged) before the next starts — so you can stack the whole game and walk away.

5 · RC operator commands

The release workflow adds two RC-lifecycle commands beyond the everyday ones. Follow the safe routine for anything destructive: back up, HALT, wait for the working task to finish, run, resume.

copy · paste# ship a ready RC by hand (operator stands in for CM) scripts/ship-rc.sh --project pgai-chomp-man --key v0.1.0 # abort an in-flight RC: backup → delete the RC branch → tasks WONT-DO → reset state scripts/unwind-rc.sh --project pgai-chomp-man --key v0.2.0 --dry-run # preview first scripts/unwind-rc.sh --project pgai-chomp-man --key v0.2.0
Note: unwind-rc cannot unwind a release that's already tagged — shipped is shipped. To re-roll a broken release, cut the next patch. And the scripts do exactly what you tell them; they don't guard against acting on a task an agent is working — HALT and wait first.

Everyday commands (same as any project)

show … --key v0.1.0
print an RC / bug / priority status
reset … --key v0.2.0
send a version back to BACKLOG to rebuild (HALT first)
close … --key BUG-0001
resolve a bug by hand
wontdo … --key PRIORITY-0001
abandon an enhancement
