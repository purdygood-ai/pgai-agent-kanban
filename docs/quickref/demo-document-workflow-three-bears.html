The document workflow — pgai-three-bears

Drop a requirement, walk away, come back to a versioned, published artifact.

This walkthrough runs an autonomous document build end to end: you drop a plain-text requirement into a project, and the agent chain plans it, writes it, and publishes a versioned artifact — no further input. Every command below is copy-paste ready and scoped to the managed demo project pgai-three-bears.

1 · How the kanban uses workflows

A project declares a workflow type in its project.cfg. The same agent chain — PM, WRITER, TESTER, CM (and CODER, for software) — runs every project, but the lifecycle they follow is set by the workflow type. The kanban ships two:

release

Software. Drives a git release-candidate branch lifecycle (open RC → code → test → ship + tag). The managed demo project is pgai-chomp-man.

document this demo

Prose deliverables. Drives off the requirement's version number and publishes a versioned artifact file. The managed demo project is pgai-three-bears.

You drive both the same way: drop a requirement file, and the chain runs on its own. The operator's whole job is dropping requirements and occasionally glancing at the dashboard.

2 · The document workflow, specifically

In the document workflow, each requirement targets a semantic version (v0.0.1, v0.0.2, …). WRITER produces the prose; CM finalizes it and publishes a versioned copy to:

projects/pgai-three-bears/artifacts/v<semver>-<artifact-name>.<ext>

The artifact name and output format come from the requirement's ## Artifact Name and ## Output Formats fields. Two properties make it powerful:

  • Versioned snapshots. Each version is its own file in artifacts/. They accumulate — v0.0.1, v0.0.2, … — so you keep a full history of the document as it evolves.
  • Extend, don't just create. A later version can instruct WRITER to read the prior artifact and modify it (add a scene, remove a character), rather than write from scratch. The requirement file says which prior version to start from.

It has full metrics and wall-time parity with the release workflow — the same dashboards (progress, logs, metrics) work identically.

3 · The example: a bedtime story that evolves

We'll build a children's bedtime story and then evolve it across versions — the exact arc that exercises every document-workflow capability:

v0.0.1
The Three Bears Bedtime Story — create from scratch
v0.0.2
A Friendly Fox at the Stream — read v0.0.1, add a scene
v0.0.3
Add an interior scene — extend again
v0.0.4
Remove the fox & stream scene — edit down
v0.0.5
Bring the fox back coherently — re-extend

Each is a single Markdown requirement file. The whole demo is just dropping these in order.

4 · Run it

1

Drop the first requirement

Deposit the v0.0.1 requirement into the project. intake.sh is a dumb router — it places the file by name; it doesn't validate contents.

copy · pastescripts/intake.sh --project pgai-three-bears \ v0.0.1-three-bears-story-20260604.md

The filename's vX.Y.* prefix routes it to projects/pgai-three-bears/requirements/. That's the only "command" the build needs — the chain takes it from here on the next cron wake.

6

Evolve the document — drop the next version

This is the part that makes it a workflow, not a one-shot. The v0.0.2 requirement tells WRITER to read the v0.0.1 artifact and add a scene. Same command, next file:

copy · pastescripts/intake.sh --project pgai-three-bears \ v0.0.2-three-bears-friendly-fox-at-the-stream.md # then v0.0.3, v0.0.4, v0.0.5 the same way — each reads the prior artifact scripts/intake.sh --project pgai-three-bears v0.0.3-bears-add-interior-scene.md

Drop them one at a time. The Active-RC gate serializes them — v0.0.1 fully publishes before v0.0.2 starts — so you can stack the whole arc and walk away.

5 · If a version needs a re-run

Not happy with a version, or it went sideways? Reset it and let the chain rebuild it. Follow the safe routine: back up, HALT, wait for any working task to finish, reset, then resume.

copy · paste# 1) pause the project (let the current task finish first) scripts/halt.sh --project pgai-three-bears # 2) reset the version's requirement — PM will re-decompose it scripts/reset.sh --project pgai-three-bears --key v0.0.2 # 3) resume — the chain rebuilds v0.0.2 on the next wake scripts/unhalt.sh --project pgai-three-bears
Note: the operator scripts do exactly what you tell them. They don't guard against resetting a task an agent is actively working — that's why you HALT and wait first. Safety lives in the routine, not the command.

wontdo … --key v0.0.2
