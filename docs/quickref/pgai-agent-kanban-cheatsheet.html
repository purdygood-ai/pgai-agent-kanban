Operator commands act on items by --key

Every command is a thin wrapper that does its one job. It refuses only when the --key can't be resolved to a single item. The operator decides; the command obeys.

Items — resolve & retire
show --project <name> --key <key>

Print an item's status / details. Read-only.

reset --project <name> --key <key> [--keep-artifacts]

Task → BACKLOG + unchecked marker, blockers cleared. Requirement → clears materialize marker (PM re-decomposes). Intake → re-opened. Refuses a WORKING task.

close --project <name> --key <key> [--state done|wont-do|superseded] [--note …] [--dry-run]

Intake item → sets terminal Status (default done). Agent task → always closed as DONE (--state is intake-only). Marker → [x].

wontdo --project <name> --key <key>

Set WONT-DO + marker [x]. The honest "abandon" verb.

delete --project <name> --key <key> [--force] [--dry-run]

Remove the item + its queue line. Guarded to terminal items.

Halt — pause dispatch
halt --project <name> # pause one project
unhalt --project <name>
halt-after --project <name> --key <key>

Graceful HALT — stops after the named work finishes (not a kill).

halt-global # pause ALL projects unhalt-global
Intake — deposit work (dumb router)
intake --project <name> <file>

Routes by filename, copies, chmod 644. Does not validate contents.

RC lifecycle
ship-rc --project <name> --key vX.Y.Z

Finalize + tag + merge a ready RC (operator stands in for CM).

unwind-rc --project <name> --key vX.Y.Z [--dry-run] [--force]

Abort an in-flight RC: backup → delete RC branch → tasks to WONT-DO → reset state. Cannot unwind a shipped tag.

--key formats self-identifying · resolve by prefix

BUG-NNNN
bug intake item
PRIORITY-NNNN
priority intake item
vX.Y.Z
requirement (version target)
ROLE-YYYYMMDD-NNN
agent task (e.g. CODER-20260626-001)

The slug tail is optional — the prefix resolves the item. Write commands refuse on an ambiguous match.

Exit codes resolving commands

0
done (or --dry-run / --help)
1
usage error / missing arg
2
ambiguous key (0 or many matches)
3
key not found
4
state mutation failed

The six states no REVIEW state

[ ]BACKLOGqueued, prereqs unmet
[W]WAITINGprereqs unmet, parked
[A]WORKINGan agent holds it now
[B]BLOCKEDagent stopped, needs input
[x]DONEcompleted
[x]WONT-DOabandoned

Agents set DONE / BLOCKED / WONT-DO. No "needs human review" state. A human can assert completion via close — rarely.

⚠ Before a destructive op reset · delete · unwind

The scripts are power tools — they do exactly what you tell them. Safety lives in the operator's routine, not in the command. Never let an agent wake mid-operation.

do this by hand first
  • Back up the project dir (tar)
  • Wait until no task is WORKING (check the dashboard)
the commands
halt --project <name> # pause first # … run reset / delete / unwind-rc … unhalt --project <name> # resume
then by hand
  • Verify on-disk state changed correctly
  • Watch the next wake on the dashboard

Intake routing & file names

filename prefix → destination

BUG-* → projects/<name>/bugs/
PRIORITY-* → projects/<name>/priority/
vX.Y.* → projects/<name>/requirements/

naming conventions
  • Requirement: vX.Y.0-<slug>.md
  • Bug: BUG-NNNN-YYYYMMDD-<slug>.md — needs the date segment and an internal # BUG-NNNN heading or intake rejects it
  • Priority: PRIORITY-NNNN-<slug>.md

Confirm the high-water before numbering:
ls projects/<p>/bugs/ | grep -oE 'BUG-[0-9]+' | sort -V | tail -1

⚠ Dispatch ≠ operator commands

Dispatch (cron) uses --agent — load-bearing, never remove:

wake-batch.sh --agent=<role> wake.sh · wake-now.sh pm-agent.sh · po-agent.sh

Operator commands use --key and never take --agent. Don't confuse the two families.

Governance read order

Every agent reads top-down:

  1. DIRECTIVES.md — top rules
  2. OVERVIEW.md — autonomy principle
  3. SOP.md — how the kanban operates
  4. README.md — entry point
  5. roles/<ROLE>.md
  6. task README.md
  7. task status.md
  8. requirement / context

The operating model

  • Files on disk are truth; git is the safety net.
  • Single-threaded by design — one RC at a time per repo (the Active-RC gate). Stacking requirements is the intended use.
  • Take a few minutes to drop requirements; the chain runs the rest autonomously.
  • CM defaults to ship — it rarely blocks (only on TESTER-BLOCKED, a missing prerequisite, or a script failure).
