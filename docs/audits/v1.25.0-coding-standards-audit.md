# Coding Standards Self-Audit — v1.25.0

## Scope

This audit walks the eleven directives in `docs/coding-standards.md`
against the pgai-agent-kanban source tree as of the v1.25.0 release
candidate. It is the credibility artifact for this release: the
framework demonstrating its own standards on itself.

The findings below are drawn from an automated audit script
(`team/scripts/audit_coding_standards.sh`) run against the release
candidate branch. The script emits per-directive verdicts using three
categories:

- **PASS** — the automated check found no violations.
- **FINDINGS** — the automated check found concrete issues; paths and
  lines are named below.
- **MANUAL-REVIEW-REQUIRED** — the directive is not fully mechanically
  checkable; supporting structural evidence is present and named
  below, and a human reviewer confirms the remainder.

The raw audit output is preserved at
`docs/coding-standards-audit-data.md`. This report is the narrative
walk-through operators and reviewers read to confirm the tree meets
the standards it publishes.

## Per-directive walk-through

### 1. Split code by concern; one implementation per operation

**Restatement.** Duplicated implementations of the same operation
drift out of sync and turn every bug into two.

**Verdict.** MANUAL-REVIEW-REQUIRED.

**Evidence.** The automated helper scanned thirty-seven shell files
under `team/scripts/lib/` and found no duplicate function names.
Whether the broader tree carries one implementation per operation at
the architectural level is not a mechanical question.

**Findings.** None from the automated check.

**Remediation status.** Manual review — the deduplication signal is
green; a reviewer confirms the architectural split at the tree level.

### 2. Keep surfaces thin

**Restatement.** CLIs, dashboards, and API handlers are adapters over
shared logic; they must not carry business rules of their own.

**Verdict.** MANUAL-REVIEW-REQUIRED.

**Evidence.** Surface thinness is a design property. The directory
structure supports it: `team/scripts/` holds CLI entry points backed
by helpers under `team/scripts/lib/`, the API router package sits
under `team/pgai_agent_kanban/api/routers/`, and dashboard pane
scripts live under `team/scripts/dashboard/` as rendering-only
consumers.

**Findings.** None from the automated check.

**Remediation status.** Manual review — a reviewer spot-checks that
CLI entry points call into helpers rather than embedding logic, API
handlers delegate to core modules, and dashboard panes only render.

### 3. Put shared code in `lib/` and import it; never copy it

**Restatement.** Copy-paste sharing forks behavior the first time
either copy changes and there is no compiler or test that catches it.

**Verdict.** MANUAL-REVIEW-REQUIRED.

**Evidence.** The structural check found `team/scripts/lib/`
populated with thirty-seven shell helpers and the
`team/pgai_agent_kanban/` Python package in place. Copy-paste
duplication across non-lib files is not statically detectable in this
tree.

**Findings.** None from the automated check.

**Remediation status.** Manual review — a reviewer spot-checks that
non-lib scripts source helpers from `lib/` rather than reimplementing
inline.

### 4. Write generic shared code

**Restatement.** Shared code shaped around a single caller becomes
private the moment a second caller needs it; write it generically
enough that a future surface can consume it unchanged.

**Verdict.** MANUAL-REVIEW-REQUIRED.

**Evidence.** Genericity is a design property, not a static-analysis
target. The intended review is that helper functions accept
parameters instead of reading globals, no helper is named after a
specific caller, and the REST API under `team/pgai_agent_kanban/api/`
consumes shared logic without duplicating it.

**Findings.** None from the automated check.

**Remediation status.** Manual review — reviewer inspects
`team/scripts/lib/` and the Python package for these properties.

### 5. No hardcoded operational values

**Restatement.** Anything an operator might change lives in a config
file with a commented `_example` companion; no hardcoded paths;
missing required config fails loud and names the key.

**Verdict.** FINDINGS.

**Evidence.** Config companions are present for the three principal
configuration files: `kanban.cfg_example`, `project.cfg_example`, and
`projects.cfg_example`. The env-bootstrap lint, however, flagged two
scripts where the approved env-bootstrap prelude is sourced after the
first non-comment reference to `PGAI_AGENT_KANBAN_ROOT_PATH`:

- `team/scripts/cm/finalize.sh` — sources the prelude at line 112 but
  the first reference to `PGAI_AGENT_KANBAN_ROOT_PATH` appears at
  line 80.
- `team/scripts/cm/open-doc.sh` — sources the prelude at line 125 but
  the first reference appears at line 96.

The first reference in each case is inside the help-text heredoc
function body, which is why the bootstrap order looks inverted:
restructuring the help-function placement relative to the bootstrap
source is a multi-line change that may affect `--help` behavior on
shells with no kanban env set.

**Findings.** Two — both named above.

**Remediation status.** Manual review — recorded as an OPEN
follow-up, not fixed in this RC. The change is behavioral-adjacent
(it touches `--help` on a bare shell) and belongs in a dedicated
ticket rather than being folded into the credibility release.

**Update — closed in v1.25.1.** The follow-up was filed as
`BUG-0087-env-bootstrap-lint-fails-help-heredoc-cm-scripts` after the
v1.25.0 gated test runners exited non-zero at the env-bootstrap lint
pre-flight for the same two scripts, and was resolved in v1.25.1: the
lint (`team/scripts/lint_env_bootstrap.py`) now ignores heredoc body
content when scanning for the first non-comment reference to
`PGAI_AGENT_KANBAN_ROOT_PATH`, so a mention inside a `--help` heredoc
above the bootstrap source no longer trips the ordering check. The
v1.25.0 finding above is retained as the historical record of what the
audit saw at the time; the finding no longer represents an outstanding
item on the current tree.

### 6. Every script and CLI entry point ships a `--help`

**Restatement.** `--help` is the contract between the operator and
the tool; a missing or stale `--help` is a silent lie to the next
person who runs it.

**Verdict.** PASS.

**Evidence.** The help-presence lint scanned seventy-one scripts
across `team/scripts` and `team/scripts/cm` with zero offenders and
zero blanket exemptions. This closes the gap the release brief
opened with: the audit of v1.24.1 flagged twenty-one scripts without
`--help`, including operator-facing entry points such as `wake.sh`,
`wake-batch.sh`, `cm/release.sh`, `cm/open-rc.sh`, and
`cm/finalize.sh`. All twenty-one now carry `--help`.

**Findings.** None.

**Remediation status.** Green.

### 7. Scaffolding and documentation ship with the feature

**Restatement.** Config examples, README sections, and generators
land in the same release as the code they describe.

**Verdict.** MANUAL-REVIEW-REQUIRED.

**Evidence.** Structural scaffolding is in place: the `docs/`
directory carries twenty-two markdown files, and a
`docker-compose.example.yaml` sits alongside the compose deployment.
Whether every new feature in this RC shipped with its documentation
is a per-diff review question.

**Findings.** None from the automated check.

**Remediation status.** Manual review — reviewer confirms the RC
diff.

### 8. Comments describe behavior, never process history

**Restatement.** Comments, docstrings, help text, log lines, and
output strings describe behavior, not process history. Do not cite
bug IDs, ticket IDs, requirement versions, or internal framework
versions in code or its runtime output.

**Verdict.** PASS.

**Evidence.** The comment-provenance lint scanned two hundred
thirty-five files across `team/scripts`, `team/scripts/lib`,
`team/scripts/cm`, and the `team/pgai_agent_kanban` package. Zero
findings.

**In-RC fix.** A small in-RC finding surfaced during the pre-audit
lint run and was corrected on this branch before the audit was
published: the help-text example in `team/scripts/cm/open-rc.sh`
carried a concrete internal task identifier, which was replaced with
a format-shaped placeholder (`CM-YYYYMMDD-NNN-open-rc`) so the
example illustrates the identifier shape without citing a specific
internal task. The lint returned PASS after the fix.

**Findings.** None outstanding.

**Remediation status.** One finding fixed in-RC; clean afterward.

### 9. Any REST API serves interactive contract docs and a versioned contract file

**Restatement.** Without published, pinnable contract docs, every
client integrates by reading server source, and every server change
is a silent break.

**Verdict.** PASS.

**Evidence.** The FastAPI application at `api/app.py` serves the
default Swagger UI at `/docs` (the `docs_url` is not overridden), the
`/docs` route is called out in the module docstring, and an
`ICD_VERSION` file is present at version 1.2.0.

**Findings.** None.

**Remediation status.** Green.

### 10. Default stack: Python 3.12+ for logic, bash for orchestration

**Restatement.** Deviate only when the requirement says otherwise and
name the reason in the ticket. A consistent default stack is what
lets one reader debug code written by another.

**Verdict.** MANUAL-REVIEW-REQUIRED.

**Evidence.** The Python version constraint in `pyproject.toml` is
`requires-python=">=3.12"`. The tree contains no non-standard
language source files (no Ruby, JavaScript, Go, Rust, or Java) under
`team/`.

**Findings.** None from the automated check.

**Remediation status.** Manual review — reviewer confirms no
logic-heavy scripts stretch `awk` or `perl` beyond one-liners, and
that any authorized deviation is documented in its ticket.

### 11. Ship Docker and compose deployment artifacts with a fail-loud entrypoint

**Restatement.** Entrypoints that swallow startup failures turn a
broken deployment into a silently degraded one; fail loud at start so
the operator sees the problem before traffic does.

**Verdict.** PASS.

**Evidence.** The Dockerfile is present at `docker/Dockerfile`, the
compose example is present at `docker/docker-compose.example.yaml`,
and `docker/entrypoint.sh` uses `set -e` and carries explicit error
exits for startup validation.

**Findings.** None.

**Remediation status.** Green.

## Closing summary

**Overall health.** The tree meets its own standards. Four directives
returned PASS with concrete mechanical evidence; six returned
MANUAL-REVIEW-REQUIRED with supporting structural evidence and no
concrete violations from the automated scans; one directive returned
FINDINGS.

Reconciliation: `total (2) = fixed-in-RC (1) + filed-as-bugs (0) +

**In-RC fixes.** One. The comment-provenance example scrub in
`team/scripts/cm/open-rc.sh` — a concrete internal task identifier

example illustrates the identifier shape without citing a real
internal task. Confirmed PASS by the comment-provenance lint after

- **D05 env-bootstrap ordering.** Two `cm/` scripts

  source the approved env-bootstrap prelude after their first

  `BUG-0087-env-bootstrap-lint-fails-help-heredoc-cm-scripts`:** the
  env-bootstrap lint was taught to skip heredoc body content, so a
  `--help` heredoc that mentions `PGAI_AGENT_KANBAN_ROOT_PATH` above

  outstanding item remains for the operator.

**Signal.** The credibility signal this release aimed for lands: the
help-presence lint is green fleet-wide (closing the 21-script gap
that motivated the release), comment-provenance is clean across two
hundred thirty-five files, and the surface-level structural checks
support the manual-review directives. The one outstanding finding is
narrow, named, and does not affect the release.
