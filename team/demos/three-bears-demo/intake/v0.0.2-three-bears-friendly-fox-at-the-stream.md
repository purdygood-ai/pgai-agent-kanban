## Overview
This is an INCREMENTAL extension of the existing v0.0.1 Three Bears bedtime story. The goal is to take
the story produced in v0.0.1 and ADD a new scene — introducing a friendly fox whom the bears meet at a
stream — while PRESERVING the existing story. This requirement is a deliberate test of whether the
document workflow can extend an existing artifact (read prior output, add to it) rather than only
generating a document from scratch.

## Brief
The substantive input the WRITER reads:

- **Starting point (READ FIRST)**: Start from version v0.0.1. The base document is the existing
  artifact `artifacts/v0.0.1-three-bears-bedtime-story.md`. WRITER must read it in full and use it as
  the basis for this version. The document workflow is versioned-snapshot: each version is a NEW file in
  `artifacts/`; WRITER reads the prior version and writes the next one with the requested changes applied.
  This version's output is a NEW file `artifacts/v0.0.2-three-bears-bedtime-story.md` that contains the
  full v0.0.1 story PLUS the new scene. The v0.0.1 file is left untouched (it remains as the prior
  snapshot).
- **What to add**: A new scene in which the three bears, out for a walk, meet a friendly fox at a stream.
  The fox is kind and curious (not a trickster — this is a gentle bedtime story). The bears and the fox
  introduce themselves and share a small, warm moment at the stream (e.g. the fox shows them where the
  water is clearest, or they watch the fish together). Keep it age-appropriate and soothing.
- **Audience and tone**: young children at bedtime; warm, gentle, simple language; soothing rhythm.
  Match the voice and reading level of the existing v0.0.1 story exactly — the new scene must feel like
  it belongs in the same story, not a stylistic departure.
- **Where the scene goes**: integrate the new scene naturally into the existing narrative flow (a new
  passage within the story, not a disconnected appendix). WRITER decides the best insertion point so the
  story still reads as one coherent piece.
- **Source material to ground in**: the existing v0.0.1 story (characters, setting, tone). Do not
  introduce facts or characters that contradict v0.0.1.
- **What to avoid**: rewriting or deleting existing passages; changing the existing characters'
  established traits; scary or sad turns (the fox is friendly); any tonal/reading-level shift from v0.0.1.
- **Length target**: add roughly 150–300 words for the new scene (a few short paragraphs); the total
  story grows by that much. Do not pad or trim the existing content.

## Deliverables
A NEW artifact file: `artifacts/v0.0.2-three-bears-bedtime-story.md`. It contains the COMPLETE v0.0.1
story (read from `artifacts/v0.0.1-three-bears-bedtime-story.md`) with the new friendly-fox-at-the-stream
scene integrated into the narrative. The v0.0.1 file is NOT modified — the versioned-snapshot model keeps
each version as its own file in artifacts/ (v0.0.1, v0.0.2, and future v0.0.3, ... accumulate as the
document's version history).

## Acceptance criteria
1. The new friendly-fox-at-the-stream scene is present in the story (the bears meet a friendly fox at a
   stream; the encounter is warm and age-appropriate).
2. The new v0.0.2 file CONTAINS the entire v0.0.1 story — WRITER started from v0.0.1 and carried its
   content forward, rather than regenerating the story from scratch (this is the core test). A diff of
   `artifacts/v0.0.1-three-bears-bedtime-story.md` vs `artifacts/v0.0.2-three-bears-bedtime-story.md`
   should show ADDITIONS (the new scene) with the original v0.0.1 passages carried through intact — NOT
   wholesale rewording/replacement of the original text.
3. The new scene is integrated into the narrative flow (reads as one coherent story, not a bolted-on
   appendix).
4. Voice, tone, and reading level of the new scene match the existing v0.0.1 story.
5. No contradictions with v0.0.1 (existing characters/setting/traits unchanged).
6. Markdown structurally valid. The deliverable is the NEW file `artifacts/v0.0.2-three-bears-bedtime-story.md`;
   `artifacts/v0.0.1-three-bears-bedtime-story.md` is unchanged.

## Notes for WRITER
- This is a VERSIONED-EXTENSION task. Step 1: read `artifacts/v0.0.1-three-bears-bedtime-story.md` in
  full. Step 2: write a NEW file `artifacts/v0.0.2-three-bears-bedtime-story.md` containing the full
  v0.0.1 story with the new scene integrated. Carry the v0.0.1 content forward faithfully — the most
  common failure mode to avoid is regenerating the whole story from the brief (which loses/alters the
  v0.0.1 wording). Start from v0.0.1's actual text and add to it. Leave the v0.0.1 file untouched.
- Match the existing voice precisely. The new scene should be indistinguishable in style from the
  original.
- Keep it gentle: the fox is friendly and the stream meeting is a calm, warm moment suitable for bedtime.

## Notes for TESTER
- The HARDEST thing to verify, and the headline check, is acceptance #2: that v0.0.2 was built FROM
  v0.0.1 (content carried forward), not regenerated from scratch. Spot-check with:
  `diff artifacts/v0.0.1-three-bears-bedtime-story.md artifacts/v0.0.2-three-bears-bedtime-story.md`
  — it should show the new scene ADDED, with the original v0.0.1 passages carried through intact. If the
  original text was reworded/replaced wholesale (even if the gist survives), that FAILS acceptance #2 —
  the versioned-extension capability (start from prior version, add to it) is what's being tested.
  Confirm the v0.0.1 file itself is UNCHANGED.
- Also verify the new fox/stream scene is actually present (#1) and integrated (#3), and that tone/reading
  level match (#4).
- This is a document-workflow verification (completeness + preservation), not a code build. No CODER
  involved.

## Note (purpose of this requirement)
This requirement tests the document workflow's versioned-extension capability: WRITER reads the prior
version (v0.0.1) from artifacts/ and writes the next version (v0.0.2) as a NEW file carrying the prior
content forward with changes applied. The artifacts/ directory accumulates the document's full version
history (v0.0.1, v0.0.2, v0.0.3, ...), each version its own file. The pass/fail signal: does
v0.0.2-...md contain v0.0.1's actual content PLUS the new scene (pass — WRITER started from v0.0.1), or
is it a from-scratch regeneration that happens to include a fox (fail — WRITER ignored the prior version)?
After this ships, artifacts/ should contain BOTH v0.0.1-...md and v0.0.2-...md.
