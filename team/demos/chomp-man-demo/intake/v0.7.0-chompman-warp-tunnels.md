## Overview

Add classic maze "warp tunnels" to chomp-man: an opening in the left and right
maze walls (at the same row) such that a character moving off one side of the
maze reappears on the opposite side at the same row. This is the wrap-around
tunnel mechanic from classic maze arcade games — applied to chomp-man's OWN
generic maze (not a reproduction of any specific copyrighted maze layout).

Unlike the v0.6.0 dragon sprites (a pure RENDER change), this is a MOVEMENT /
BOUNDARY-LOGIC change — it exercises the game's position-update and wall-collision
code, a different code path. Good coverage for the test sandbox.

## The feature
- Choose one horizontal row near the vertical middle of the maze (or reuse an
  existing corridor row that reaches both side walls) to serve as the tunnel row.
- Open the left and right maze boundary at that row (remove/zero the wall cells
  at the far-left and far-right of that row) so the corridor is continuous to
  both edges.
- When a character (the player chomper, AND each enemy) moves PAST the left edge
  on the tunnel row, it reappears at the right edge on the same row, and vice
  versa. The wrap is seamless — position wraps modulo the play-field width on
  that row.
- Both the player and the enemies use the tunnel (classic behavior: enemies can
  chase through it too).

## Constraints
- Apply to chomp-man's OWN maze. Do NOT copy a specific arcade maze layout — this
  is the generic wrap mechanic on the existing chomp-man maze.
- The wrap must respect the existing grid/cell coordinate system — a character
  exiting at the tunnel row's left edge reappears aligned on the same row at the
  right edge (no vertical drift, no off-grid position).
- Collision/wall logic elsewhere is UNCHANGED — only the tunnel row's edge cells
  become passable, and only that row wraps. All other walls still block.
- Dots/pellets: the tunnel row may or may not have pellets at the very edge
  cells — keep it consistent (either no pellet in the tunnel mouth, or a normal
  pellet that's collectible). Do not leave an uncollectible pellet that blocks
  board-clear (the game ends on board-clear — an unreachable pellet would soft-lock
  the win condition). THIS IS IMPORTANT: verify the board can still be fully
  cleared after adding the tunnel.
- Keep movement speed/AI otherwise unchanged; the tunnel is a position-wrap, not
  a speed or behavior change.

## Acceptance Criteria
Chain-verifiable:
1. The game launches and runs without error (`python3 <entrypoint>` starts; no
   exceptions). Any existing chomp-man tests pass.
2. The maze data has an opening at the left AND right boundary on the tunnel row
   (verifiable by inspecting the maze grid: the far-left and far-right cells of
   the tunnel row are passable, not walls).
3. The movement code wraps position on the tunnel row: a position moving past the
   left edge maps to the right edge (and vice versa) on the same row — verifiable
   by a unit test of the position-wrap function if one is added, or by inspecting
   the wrap logic.
4. The wrap applies to BOTH player and enemies (the same boundary-wrap code path
   handles all moving characters, or each is handled equivalently).
5. Board-clear is still achievable: no pellet is made unreachable by the tunnel
   change (no soft-lock of the win condition). If pellets exist on the tunnel
   row, they remain collectible.
6. `python3 -m py_compile` on changed files; the game's own test suite (if any)
   passes.

Operator-verifiable (NOT chain-verifiable — flag for operator):
7. OPERATOR PLAY-TEST: run chomp-man, move the chomper off the left/right edge on
   the tunnel row, confirm it reappears on the opposite side smoothly and aligned;
   confirm enemies also wrap; confirm the board can still be cleared to the
   win/end state. (Movement feel and visual correctness are the operator's
   judgment.)
