## Overview
The enemy characters in chomp-man currently render as simple colored diamonds
(rotated squares) — orange, green, cyan, red, purple. Replace the diamond shape
with a small DRAGON SILHOUETTE drawn with pygame primitives (polygons/circles),
so the enemies read as creatures rather than gems. PRESERVE each enemy's existing
color (the colors are per-enemy identity and must remain distinct).

This is a sandbox/demo enhancement to the chomp-man test project (NOT a kanban
framework change). It exercises the release workflow on a real visual-feature
task and produces nicer demo footage.

## Approach (deliberately constrained — do NOT freelance the rendering method)
- Draw the dragon with **pygame drawing primitives** (`pygame.draw.polygon`,
  `pygame.draw.circle`, `pygame.draw.lines`) composed into a small silhouette.
  Do NOT load external image/sprite asset files — keep it self-contained and
  matching the game's existing flat-shaded geometric style.
- The enemy is small (roughly one maze cell, ~20-25px). The dragon must be a
  READABLE SILHOUETTE at that size — a suggestive shape, not a detailed
  illustration. Aim for: a rounded/tapered body, a pair of wing shapes
  (triangles), and a small head and/or tail point — enough that it reads as
  "winged creature / dragon," not "diamond."
- Fill the dragon in the enemy's EXISTING color (each enemy keeps its current
  color — orange, green, cyan, red, purple). Color is per-enemy identity; do not
  collapse them to one color.
- Keep it centered and sized to the same cell footprint the diamond occupied, so
  collision/movement/maze alignment are UNCHANGED (this is a render-only swap at
  the draw site; do not alter enemy position, hitbox, speed, or AI).

## Constraints
- Render-only change: locate where the enemy diamond is currently drawn and
  replace the shape drawing. Do NOT change enemy movement, collision, spawn,
  count, or game logic.
- Preserve per-enemy color.
- Preserve cell footprint / alignment (the dragon occupies the same space the
  diamond did).
- The game must remain fully playable (runs, enemies move, collisions work,
  lives/score unaffected).
- If a facing/direction is easy to honor (dragon points the way it moves), that's
  a nice-to-have, NOT required — do not over-engineer; a static silhouette is
  acceptable for v0.6.0.

## Acceptance Criteria
Chain-verifiable:
1. The game launches and runs without error (`python3 <entrypoint>` starts; no
   exceptions on the render path). If there's a headless/smoke test, it passes.
2. The enemy draw code uses pygame primitives (polygon/circle/lines) — NOT a
   loaded image asset — verifiable by grep of the changed draw function.
3. Each enemy is still drawn in its own distinct color (the per-enemy color
   values are preserved in the code — orange/green/cyan/red/purple still present
   and mapped per enemy).
4. Enemy position/hitbox/movement code is UNCHANGED (diff shows changes confined
   to the draw/render function; no edits to movement/collision/AI).
5. `python3 -m py_compile` on changed files; any existing chomp-man tests pass.

Operator-verifiable (NOT chain-verifiable — flag for operator):
6. OPERATOR VISUAL CHECK: run chomp-man and confirm the enemies READ as
   dragons/creatures (a glance says "monster," not "gem"), each still
   distinguishable by color, sized correctly in the maze. (Game graphics are a
   blind-fix category — the chain can verify the code runs and draws, but only
   the operator can judge whether it looks like a dragon. Expect a possible
   second iteration to refine the silhouette.)
