## Overview
Build the playfield and the player. This release renders a maze and places Chomp Man (the
player character) in it, moving under keyboard control. No dots, no dragons, no scoring yet —
just a maze you can move the hero around in, with walls that block movement.

## Goals
1. A maze rendered on screen: walls and open corridors, in a fixed grid layout.
2. Chomp Man rendered as a distinct character (a chomping wedge — the open-mouthed circle is
   fine and on-theme, but give it Chomp Man's own color/identity).
3. Keyboard movement on the NON-STANDARD scheme (case-insensitive):
   - `A` / `a` = up
   - `Z` / `z` = down
   - `J` / `j` = left
   - `L` / `l` = right
4. Wall collision: Chomp Man cannot move through walls; movement into a wall is blocked.
5. Smooth, grid-aligned movement at a steady frame rate.

## Design / Theming (Chomp Man identity)
- The control scheme is deliberately A/Z/J/L (left-hand A/Z, right-hand J/L) — two-handed, and
  it intentionally leaves `K` unbound (reserved for a future special-effect key). Do NOT use
  WASD or arrow keys; the custom scheme is part of the game's identity and a literal spec to
  follow.
- The maze should have its own look (color scheme, wall style) — not a carbon copy of any
  existing game's maze. Pick a clean, readable aesthetic.
- Chomp Man faces the direction of travel (mouth opens toward movement) if feasible; nice-to-
  have, not required for this release.

## Deliverables
- Updated `pgai-chomp-man.py` (and any supporting modules the coder chooses to create, e.g. a
  maze definition, a player class):
  - A maze data structure (a grid; walls vs open cells) and its rendering.
  - Chomp Man drawn in the maze, starting at a sensible open cell.
  - Keyboard handling for A/Z/J/L (case-insensitive) mapped to up/down/left/right.
  - Wall-collision logic preventing movement into walls.
  - A steady game loop (fixed FPS, e.g. 60) with clean quit handling (Q / window close).

## Acceptance criteria
1. `python3 -m py_compile` passes for all changed/added .py files.
2. The maze renders (walls + corridors) and Chomp Man is visible in it (manual visual check).
3. Movement keys work case-insensitively: pressing `a` OR `A` moves up; `z`/`Z` down; `j`/`J`
   left; `l`/`L` right. (Verify the key→direction mapping in code is case-insensitive — e.g.
   handles both `pygame.K_a` with/without shift, or lowercases input.)
4. Chomp Man cannot move through walls (collision blocks the move).
5. `K` is NOT bound to any movement (reserved). Arrow keys and WASD are NOT bound to movement.
6. The game maintains a steady frame rate and quits cleanly on Q / window close.

## Suggested Decomposition

- CODER-2: Chomp Man entity + grid-aligned movement + the A/Z/J/L case-insensitive input map.
- CODER-3: wall-collision logic + integrate into the game loop.

## Notes for CODER
The A/Z/J/L scheme is a literal requirement — implement exactly that, case-insensitive, and
leave K unbound. Keep the maze in a data structure (e.g. a list of strings or a 2D grid) so
later releases can place dots, dragons, pellets, and fruit into the same grid. Grid-align
movement so collision and (later) dot-eating are simple cell checks. This is the foundation
for everything after — structure the maze/player code to be extended, not rewritten.
