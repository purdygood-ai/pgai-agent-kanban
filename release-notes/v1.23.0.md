# Release Notes — v1.23.0

## Status
released — PASS, no concerns (the launch tag's verification carried
zero findings; both gated suites green at 2,147 unit / 193
integration)

## Summary
v1.23.0 is the second public release and the consolidation of
everything since v1.0.0: the pluggable workflow architecture, the
operator REST API with its browser dashboard, the OVERWATCH observer
and PO intake agent, a unified fail-loud environment contract across
every entry point, and a third shipped workflow type — testing-only —
certified end-to-end with zero operator interventions. Every release
between v1.0.0 and this tag was decomposed, implemented, verified,
and shipped by the framework itself.

## What Shipped

- **Pluggable workflow types.** Workflows are now plugins: a
  directory with a manifest and eight hooks. The engine queries
  capabilities (version semantics, git mode, roster, finalize), never
  type names, and fails closed on unknown types. A generator
  scaffolds new types; `validate-workflow.sh` gates them; a
  ready-status flip is the human gate. `release` and `document` are
  themselves plugins; custom types survive upgrades by contract.
- **testing-only — the third workflow type.** Audits something
  already shipped: a read-only, detached-worktree verification of any
  project's tag. Label versioning (nothing consumed), a pm+tester
  roster, a report as the finalize product, and machinery closure of
  the intake item. Genuine findings are filed on the target project's
  own lane with full provenance. Certified end-to-end — selection to
  report to closure — with zero operator interventions, during which
  it found and filed real defects in the project it was auditing.
- **Operator REST API + browser dashboard.** Seventeen operations
  behind a loopback FastAPI service with Swagger docs, universal
  `dry_run` support and a `warnings` field on every response,
  strict-validation 422s proven to fire before any subprocess, and a
  structured read layer (board, project metadata, metrics/costs as
  JSON, log access with server-side path confinement, trace
  index/fetch). API parity with the CLI is lint-enforced: a script
  growing a flag fails the suite until the API matches. The browser
  dashboard covers board, attention, inspector, metrics, logs, and a
  gated operate surface. Both ship labeled experimental.
- **OVERWATCH + PO.** OVERWATCH runs an hourly zero-token health
  sweep and a daily LLM deep-clean, and wakes immediately when any
  agent blocks (storm-collapsed, self-trigger-guarded, HALT-gated).
  PO turns a plain-language brief into a formal requirements doc
  under a nine-rule authoring method with an explicit assumptions
  ledger.
- **Fail-loud environment contract.** Every script — shell and
  Python — establishes its environment through one bootstrap:
  self-locate, source the install's `shell-env`, fail loudly with the
  exact path if the root is still unset. Explicit operator exports
  always win. Enforced by lint on both runtimes in every gated run.
- **Provider abstraction hardened.** One config line switches
  providers; per-role model overrides; the resolved model is
  wake-stamped into each task's status as an execution record.
- **Release engine resilience.** The release step is resumable at
  its checkpoint boundaries after interruption or damage;
  cancellation drills are part of the standing verification; the
  CHANGELOG and ICD freshness gates hold byte-exactness on the
  released line while tolerating exactly the legitimate in-flight-RC
  shapes.
- **Two-repo law.** Live installs import only from themselves and
  fail loudly on their own brokenness; repository checkouts are data,
  never a runtime dependency. Codified in the SOP, enforced in the
  dashboard's import paths, and testable in one move: flipping any
  dev tree's branch changes nothing about live behavior.

## Breaking Changes

- **Environment contract.** Scripts no longer run with a silently
  defaulted root. On a standard install the bootstrap finds
  `shell-env` automatically and nothing changes for operators;
  invoking scripts from a bare checkout (no install) now fails loudly
  by design.
- **Model configuration.** Provider models live in per-provider
  sections (`[models.claude]`); a flat `[models]` block from v1.0.0
  configs is ignored — move values into the provider section when
  upgrading.
- **Workflow types are plugins.** Any external tooling that assumed
  a fixed two-type world should consult the plugin manifests; unknown
  types are refused rather than defaulted.

## Upgrade Notes

From v1.0.0: run `upgrade.sh` from the live install (it preserves
projects, registry, and secrets, and overlay-copies without
deleting — operator-authored workflow types survive). Answer the
crontab prompt to match your active scheduler. The upgrader requires
its target explicitly when invoked from a dev tree — the environment
contract applied to the upgrader itself.

## Bug Fixes

Consolidated across the arc: plugin-era capability fixes through the
entire engine (registration, eligibility, filename handling, version
ceilings, task-state transitions, graph materialization, folder
recycling), API lifecycle and contract hardening, dashboard
truncation and import-path integrity, release-gate tolerance
corrections, and operator-tool recipes brought current with the
selection model. Defects are tracked on the internal ledger and
described by pattern in the CHANGELOG.
