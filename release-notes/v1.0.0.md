# pgai-agent-kanban — v1.0.0

**An autonomous multi-agent build system that ships its own releases.**

Drop a requirements document. Walk away. Come back to a tagged release.

This is the v1.0.0 release of pgai-agent-kanban: feature-complete, code-verified,
and self-hosting — the system builds and ships its own releases. It is **released
rather than launched** — no announcement, no support commitment — for a small
circle of users and for anyone who finds it. The architecture is settled and the
mechanics are exercised end-to-end. Expect the occasional rough edge and treat it
as feedback: this is an early release meant to be used and improved in the open,
not a finished product with a warranty.

This document describes the system **as it stands now**. It is not a changelog —
the development history (hundreds of self-built releases) lives in the archive,
not here. What follows is a clean statement of what the system is at v1.0.0.

---

## What it is

pgai-agent-kanban orchestrates a chain of specialized LLM agents through a
single-threaded, file-driven pipeline. You hand it a structured requirements
document; it decomposes the work into discrete tasks, executes them, verifies the
result, and ships it — unattended, on a single VPS, driven by cron and observed
through a tmux dashboard.

The operator's job is to drop requirements and occasionally glance at the
dashboard. Everything between "requirement dropped" and "release tagged" is
autonomous.

It is **provider-abstracted**: which LLM executes a task is a runtime config
value, not a property of the work. Claude is the supported production provider; a
second provider lane (codex) is present and has run work in testing, but ships
**experimental** for this release.

---

## The agent chain

A project is driven by a chain of role-specialized agents, each woken on its own
cron cadence:

- **PM** — decomposes a requirements document into discrete, ordered tasks.
- **CODER** — implements software tasks.
- **WRITER** — produces prose/document deliverables.
- **TESTER** — verifies the result, categorizes findings, and files bugs. It
  reports; it does not block on found problems (it blocks only on a pre-flight
  failure).
- **CM** — the release manager. The sole agent that touches the git remote.
  Defaults to shipping; refuses only on a TESTER-blocked pre-flight, a missing
  prerequisite, or a script failure.

(A human-invoked **PO** role and an automated **OVERWATCH** recovery role are
scaffolded in the system and planned for a later release; they are not part of the
autonomous chain today.)

## Core principles

1. **Six states, no review gate.** Work is BACKLOG, WAITING, WORKING, BLOCKED,
   DONE, or WONT-DO. Agents set DONE / BLOCKED / WONT-DO. There is no "needs human
   review" state — the human steers through the requirement, not through a queue.
2. **Files on disk are the source of truth; git is the safety net.** State lives
   in files (task `status.md`, queue markers, `release-state.md`). Git holds the
   release lane and the recovery net.
3. **Single-threaded by design.** One release candidate at a time per repository,
   serialized by the Active-RC gate. This is the safety mechanism that makes
   stacking requirements safe: requirement N fully ships before requirement N+1
   begins. Stacking is the intended way to use the system, not a risk to manage
   around.
4. **Local-only git for non-CM agents.** CODER, WRITER, and TESTER never push,
   pull, or fetch; each task runs in its own isolated git worktree. CM is the only
   agent that interacts with the remote, and only at release time.
5. **The requirement encodes the human's decision; agents execute it.** File
   allowlists, must-not-modify guards, and acceptance tripwires in the requirement
   steer the chain.

## Two workflows

- **release** — software. Drives a git release-candidate branch lifecycle: PM
  decomposes, CM opens an RC branch, CODER implements, TESTER verifies, CM ships
  (merge + tag). Projects can run in an isolated branch lane via a configurable
  `branch_prefix`, so a managed repo's human branches stay untouched.
- **document** — prose deliverables. Drives off the requirement's semantic
  version and publishes a versioned artifact to
  `projects/<name>/artifacts/v<semver>-<name>.<ext>`. Each version is its own
  snapshot; a later version can read the prior artifact and extend it. Full
  metrics and wall-time parity with the release workflow.

## The operator surface

Operators interact through a uniform command vocabulary. Every command takes a
self-identifying `--key` (`BUG-NNNN`, `PRIORITY-NNNN`, `vX.Y.Z`, or
`ROLE-YYYYMMDD-NNN`), resolved by prefix with ambiguity refused on write
operations. Commands reject unknown flags rather than silently ignoring them, so a
mistyped flag fails loudly instead of running with a wrong default:

- **Inspect / retire:** `show`, `reset`, `close`, `wontdo`, `delete`
- **Pause dispatch:** `halt` / `unhalt`, `halt-after`, `halt-global` /
  `unhalt-global`
- **Deposit work:** `intake` (routes a requirement, bug, or priority file by name)
- **Release lifecycle:** `ship-rc`, `unwind-rc`
- **Projects & config:** `create-project` / `add-project` / `remove-project`,
  `set-version-ceiling`, `switch-provider`, and project / config import-export

These commands are thin entry points over a shared operations library
(`pgai_agent_kanban/ops/`) — one implementation per operation, so the same logic
backs the CLI today and is positioned for a programmatic API in the future. The
library owns the safety invariants (atomic writes, refuse-on-WORKING for reset,
intake validation), so they hold for every caller.

The scripts are precise power tools: they do exactly what they are told, and they
document exactly what they do. Operational safety lives in the operator's
procedure (back up, halt, verify, resume), not in the commands second-guessing the
operator.

## Observability

A tmux dashboard renders the live state across dedicated windows: a main overview,
a per-agent visibility board, attention items, git status, per-version metrics
(wall-time and token cost), agent logs, and the upcoming wake schedule. All
state is read from the files on disk; git is read only for genuinely git-shaped
information (the git-status panes and shipped-release detection).

---

## Layout

- **Dev tree** — where the source is edited and committed; carries a `team/`
  source prefix.
- **Live install** — where cron-driven agents run; `install.sh` copies the
  framework into the install root, dropping the `team/` prefix.
- A project is registered with one `create-project` invocation. The framework
  itself is managed as an ordinary registered project (self-build) — nothing in
  the code knows or cares that the managed repository happens to be its own
  source.

## Status at v1.0.0

- The agent chain, both workflows, the operator surface, the shared operations
  library, and the dashboard are implemented and exercised end-to-end.

  the unit level.
- Claude is the supported production provider; the codex provider lane is present
  but experimental.
- The reference documentation ships with this release: README, ARCHITECTURE, the
  operator command reference (`docs/operator-commands.md`), the quick-reference
  guides (`docs/quickref/`), and the operational guides (dashboard, disk hygiene,
  troubleshooting, and project configuration).
- Planned for after v1.0.0: the PO and OVERWATCH roles, broader provider support,
  and a programmatic API over the shared operations library.

This is a space to build with confidence and verify against the running system —
which is exactly how it was built.
